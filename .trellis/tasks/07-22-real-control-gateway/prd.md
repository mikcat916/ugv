# 接入真实控制网关

## Goal

让 Web 管理后台创建的非单机器人控制命令能够通过配置的真实控制网关下发，并把成功、失败和返回内容可靠地记录到 `control_commands` 表中，避免当前“配置了地址但仍固定报未接入”的状态。

## Background

- `POST /api/control/commands` 已要求管理员登录，并会先创建一条 `pending` 命令记录。
- 单机器人命令已经通过 TCP 控制服务执行，支持速度、停车、急停和连通性检查。
- `sensor`、`device`、`network_channel`、`onboard_unit`、`cluster_node` 等目标只检查数据库记录是否存在。
- 当 `CONTROL_GATEWAY_URL` 未配置时，后端返回 `502`；即使已经配置，当前仍固定返回“真实控制网关暂未接入”。
- 成功命令应更新为 `success` 并保存 `response_json`；失败命令应更新为 `failed` 并保存错误信息。
- `cluster_node + node_exit` 成功后，现有代码会把节点状态改为 `disconnected` 并清空 `joined_at`。

## Requirements

- 项目内新增一个可独立启动的最小 HTTP 控制网关，不依赖项目外已有服务。
- 网关默认使用安全模拟模式；只有显式启用真实转发配置后才允许调用下游设备接口。
- 第一版只接受 `cluster_node` 目标，以及 `connectivity_test`、`node_exit` 两种命令。
- `connectivity_test` 通过集群节点关联机器人的 TCP 控制服务执行 `ping`。
- `node_exit` 先通过 TCP 控制服务确认停车成功，再允许后端把集群节点标记为已退出。
- 模拟模式必须返回 `executed=false`；后端记录为 `simulated`，不得执行节点退出副作用。
- 保留现有单机器人 TCP 控制链路，不改变 `/api/robot-control/*` 和直接机器人命令行为。
- 非单机器人控制命令必须使用明确、可测试的网关协议，不能伪造成功。
- 网关地址缺失、连接失败、超时、非成功状态和错误响应必须返回可理解的错误。
- 只有网关明确确认成功后，命令才能标记为 `success` 并执行数据库副作用。
- 所有外部返回内容在写入数据库和返回浏览器前需要限制大小并验证基本格式。
- 网关认证信息只能通过环境变量配置，不能写入源码、日志或普通 API 响应。
- 为成功、拒绝、连接失败、超时、格式错误和缺少配置增加自动测试。
- 更新 `.env.example` 和维护文档，说明网关配置、协议和本地测试方法。

## Acceptance Criteria

- [x] 配置有效网关后，受支持的非机器人命令会发送到网关，并将网关确认结果返回给管理员。
- [x] 第一版仅支持集群节点的连通性检测和退出，其他目标或命令返回明确的 `422` 错误。
- [x] 连通性检测向关联机器人的控制服务发送 `ping`，仅在收到 `pong` 后报告执行成功。
- [x] 节点退出先发送停车命令，只有收到成功确认后才更新数据库节点状态。
- [x] 模拟模式不会发送 TCP 指令，也不会修改集群节点状态。
- [x] 未配置网关时返回明确错误，命令记录为 `failed`。
- [x] 网关连接失败、超时、拒绝或返回异常内容时不执行成功副作用，命令记录为 `failed`。
- [x] 网关明确成功时命令记录为 `success`，并保存经过限制的返回内容。
- [x] `cluster_node + node_exit` 仅在网关确认成功后更新节点状态。
- [x] 普通用户不能通过控制接口下发命令。
- [x] 自动测试不连接真实设备或互联网，并覆盖主要成功和失败路径。
- [x] 现有后端和工具测试继续通过。

## Out of Scope

- 修改单机器人 TCP 控制协议。
- 自动发现真实网关地址。
- 在单元测试中向真实车辆发送命令。
- 重新设计 Web 管理页面或数据库表结构。
- 传感器、普通设备、网络通道、机载单元和编队级命令。
- 新增第二套机器人底层控制协议。
